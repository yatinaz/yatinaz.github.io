Architecture

End-to-end RAG pipeline. Ingestion (top row): PDFs are parsed with pdfminer and Camelot, split into typed chunks, embedded with all-MiniLM-L6-v2, and stored in a persistent ChromaDB collection. Query path (bottom row): a user question is embedded and matched against the index; the top-k chunks are passed to Groq's llama-3.3-70b for grounded answer generation with inline citations.

Design Decisions

Chunking strategy. PDFs are parsed with pdfminer for text and Camelot for tables, then split into typed chunks (text, table, figure) with overlap-free boundaries at detected section headings. A garble filter discards chunks whose space-to-character ratio exceeds a threshold — this eliminates corrupted figure captions and ligature-encoded garbage that would poison the retrieval index. Chunks below 30 tokens are dropped; chunks above 512 tokens are split at sentence boundaries.

Embedding model choice. all-MiniLM-L6-v2 (384 dimensions) was chosen over larger models for speed and memory footprint — the full corpus embeds in under 30 seconds on CPU. ChromaDB uses cosine similarity in a persistent on-disk collection so the embedding step runs once; subsequent queries hit only the index.

Eval set construction. An initial set of 23 LLM-generated candidate questions was audited against strict criteria: remove exact duplicates, remove locating questions where the answer is just a filename or section name, remove metadata questions answerable without retrieval. Nine questions were dropped; four replacements were generated via Groq targeting factual and comparative types only. The final 18-question set has 3 questions per paper and excludes trivially answerable questions — making it a conservative upper bound on retrieval quality.

Grounded generation. The generator constructs a context block from the top-k retrieved chunks, each prefixed with source file, section, and page range. The system prompt instructs the model to cite sources for every factual claim using the [Source N: ...] convention and to explicitly state when the context is insufficient. Temperature is set to 0.1 to minimise hallucination.

P@5 vs P@3 gap. The 16.7 pp gap between P@3 (0.444) and P@5 (0.611) shows that the relevant chunk typically lands in positions 4–5 for misses — the retriever is finding the right document but not always ranking the exact chunk first. The comparative question type is harder (P@3=0.286) because comparative answers often span multiple chunks, but evaluation uses a single relevant chunk ID per question.
